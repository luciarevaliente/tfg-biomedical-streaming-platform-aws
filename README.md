## Overview

Distributed cloud architecture for the near real-time ingestion, processing, persistence and monitoring of biomedical data from wearable devices. The use case is the continuous monitoring of hospital patients in non-critical areas.

The system receives continuous streams of physiological signals (ECG, EDA, BVP, TEMP, RESP, ACC, EMG), processes them with end-to-end latencies in the order of seconds, and stores both raw and processed records for traceability and analysis.

**Primary SLO:** P95 pipeline latency < 10 s, measured from Kinesis arrival to end of Lambda processing, with a data loss rate below 1%.

---

## Architecture

Fully serverless, event-driven architecture on AWS, structured in five functional blocks:

```
Local Simulator → Kinesis Data Streams → AWS Lambda → S3 (raw)
                                                    → DynamoDB (processed)
                                                    → SQS DLQ (failed events)
                                         CloudWatch + CloudTrail (monitoring & audit)
```

| Block | Service | Role |
|---|---|---|
| Data generation | Python simulator (local) | Reproduces wearable device streams at real WESAD sampling rates |
| Ingestion | Amazon Kinesis Data Streams (5 shards) | Ordered buffering by `subject_id#sensor_type` partition key |
| Processing | AWS Lambda (Python 3.11, 128 MB) | Validation, unit conversion, timestamp enrichment |
| Raw persistence | Amazon S3 | Event-level JSON batches for traceability and reprocessing |
| Processed persistence | Amazon DynamoDB (PAY_PER_REQUEST) | Per-event records with latency metrics, keyed by `subject_id#sensor_type` + UUID |
| Error handling | Amazon SQS (DLQ) | Failed events routed for manual inspection |
| Monitoring | Amazon CloudWatch | Custom metrics under `BiomedicalPipeline` namespace, alarms on key indicators |
| Audit | AWS CloudTrail | Management event log for resource access traceability |
| Cost control | AWS Budgets | Automated alerts at 80% and 100% of a $10/month budget |
| Infrastructure | Terraform ≥ 1.0 | Fully declarative IaC, reproducible deployments |

Infrastructure state is stored in an S3 backend (`tfg-biomedical-terraform-state`) with DynamoDB locking (`tfg-biomedical-terraform-locks`).

## Data model

Sampling frequencies are taken from the [WESAD dataset](https://archive.ics.uci.edu/dataset/465/wesad+wearable+stress+and+affect+detection) (UCI ML Repository) as a structural reference. The simulator generates synthetic events at those exact rates — no real device data is used.

| Device | Sensor | Signal | Rate |
|---|---|---|---|
| RespiBAN | ECG, EDA, EMG, TEMP, RESP, ACC | Various physiological | 700 Hz |
| Empatica E4 | BVP | Pulse volume (PPG) | 64 Hz |
| Empatica E4 | ACC | Acceleration | 32 Hz |
| Empatica E4 | EDA, TEMP | Electrodermal activity, skin temperature | 4 Hz |

Two record types travel through the pipeline:

- **Raw record** — event exactly as generated by the simulator, written to S3 in JSON batches of up to 500 events per Lambda invocation.
- **Processed record** — enriched with three timestamps and three latency metrics, persisted individually to DynamoDB.

### Latency measurement model

All timestamps are generated within the AWS domain to avoid the clock skew between the local simulator and AWS (~74 ms offset measured experimentally).

| Point | Field | Source |
|---|---|---|
| T0 | `kinesis_arrival_ts` | `approximateArrivalTimestamp` assigned by Kinesis |
| T1 | `processing_start_ts` | `time.time()` at the start of per-event processing in Lambda |
| T2 | `processing_end_ts` | `time.time()` immediately before the DynamoDB batch write |

| Metric | Formula | Description |
|---|---|---|
| `kinesis_to_lambda_ms` | (T1 − T0) × 1000 | Time buffered in Kinesis before Lambda consumed the batch |
| `processing_latency_ms` | (T2 − T1) × 1000 | Internal Lambda time: decode, validate, enrich |
| `pipeline_latency_ms` | (T2 − T0) × 1000 | **End-to-end latency — primary SLO metric** |

> **Known limitation:** the DynamoDB `batch_write` (~277 ms average duration per invocation) is excluded from `pipeline_latency_ms`. This is a systematic underestimation documented in the thesis appendices. It does not affect the SLO assessment because the Kinesis buffer time dominates by at least one order of magnitude in all scenarios.

## Experimental scenarios

| Scenario | Setup | Rate | Duration | SLO |
|---|---|---|---|---|
| Base | 1 subject — Empatica E4 (EDA + TEMP) | ~8 ev/s | 3 × 5 min | P95 < 10 s, loss < 1% |
| Sustained | 2 subjects — full Empatica E4 (BVP, ACC, EDA, TEMP) | ~208 ev/s | 3 × 5 min | P95 < 10 s, loss < 1% |
| Peak | 1 subject — full E4 + RespiBAN (all sensors, 700 Hz) | ~4,304 ev/s | 3 × 30 s | Degradation quantification |

Each scenario runs 3 consecutive repetitions with a 60-second interval between them. The `run_id` field (1, 2 or 3) identifies each repetition in DynamoDB.

**Key results:** base and sustained scenarios meet all SLOs (P95 of 1,037 ms and 1,074 ms respectively, 0.000% data loss). The peak scenario reveals Kinesis shard throttling as the limiting factor (14.195% loss, P95 up to 197,602 ms), not Lambda — `processing_latency_ms` is 0 ms across all scenarios and percentiles.

## Estimated cost

Single experimental session (~2 hours), region `eu-west-1`, all three scenarios + nine repetitions:

| Service | Cost |
|---|---|
| Amazon Kinesis Data Streams | $0.095 |
| AWS Lambda | $0.00 (Free Tier) |
| Amazon DynamoDB | $0.418 |
| Amazon S3 | $0.020 |
| Amazon CloudWatch | $0.00 (Free Tier) |
| **Total** | **~$0.533** |

In a continuous production deployment, the dominant cost is Kinesis in PROVISIONED mode: ~$2.04/day (5 shards × 24 h × $0.017/shard-hour). Lambda and CloudWatch remain within Free Tier at these scales. See Appendix 4 of the thesis for a full breakdown and production cost projection.

## How to run

### Prerequisites

- [Terraform ≥ 1.0](https://developer.hashicorp.com/terraform/install)
- [AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2.html) configured with credentials for an AWS account (`eu-west-1` region)
- Python ≥ 3.11 with `boto3` installed
- An S3 bucket named `tfg-biomedical-terraform-state` and a DynamoDB table named `tfg-biomedical-terraform-locks` already created in your account (used for Terraform remote state — create them once manually or via the AWS console before the first `terraform init`)

### 1. Deploy the infrastructure

```bash
cd infrastructure/terraform

# Initialise Terraform (downloads providers, configures S3 backend)
terraform init

# Preview the changes
terraform plan

# Deploy all AWS resources
terraform apply
```

This creates the Kinesis stream, Lambda function, S3 bucket, DynamoDB table, SQS DLQ, CloudWatch dashboards and alarms, CloudTrail trail, IAM roles, and budget alerts.

### 2. Verify the pipeline (smoke test)

Run a single end-to-end test before launching any scenario. The script sends one event to Kinesis and polls DynamoDB for up to 60 seconds until Lambda processes it.

```bash
cd src/smoke_test
pip install boto3
python smoke_test.py
```

Expected output: the processed record appears in DynamoDB with `pipeline_latency_ms > 0` and the three latency fields consistent.

### 3. Run an experimental scenario

```bash
cd src/data-generator
pip install -r requirements.txt   # installs boto3

# Available scenarios: base | sustained | peak
python simulator.py --scenario base
python simulator.py --scenario sustained
python simulator.py --scenario peak
```

The simulator runs all 3 repetitions automatically with a 60-second cooldown between them. It logs events sent locally (used later for integrity verification) and writes directly to the Kinesis stream `tfg-biomedical-dev-stream`.

> **Note for the peak scenario:** at ~4,304 ev/s, the current 5-shard configuration will produce Kinesis throttling. This is the documented behaviour studied in the thesis. Run it to reproduce the peak results, not to validate production readiness.

### 4. Analyse the results

After a scenario completes, query DynamoDB and generate all figures and CSVs:

```bash
cd src/analysis
pip install boto3 pandas matplotlib numpy scipy

python analysis.py --scenario base
python analysis.py --scenario sustained
python analysis.py --scenario peak
```

Output is written to `experiments/results_<scenario>/` (CSVs and PNG figures). The `fig4_peak_timeline.png` is only generated for the peak scenario.

### 5. Tear down the infrastructure

```bash
cd infrastructure/terraform
terraform destroy
```

> Remember to empty the S3 bucket before running `destroy` — Terraform cannot delete non-empty buckets.

## AWS service configuration reference

| Component | Parameter | Value |
|---|---|---|
| Kinesis Data Streams | Shards | 5 (PROVISIONED) |
| Kinesis Data Streams | Retention | 24 hours |
| Lambda | Runtime / memory | Python 3.11 / 128 MB |
| Lambda ESM | `batch_size` | 500 |
| Lambda ESM | `parallelization_factor` | 10 (→ max 50 concurrent invocations) |
| Lambda ESM | `bisect_batch_on_function_error` | enabled |
| DynamoDB | Billing mode | PAY_PER_REQUEST |
| DynamoDB | Partition key | `subject_id#sensor_type` |
| DynamoDB | Sort key | `event_id` (UUID) |
| DynamoDB | Write method | `batch_writer` (500 writes → ~20 batch calls) |
| S3 | Object key | `raw/{timestamp}_{request_id}.json` |
| CloudWatch | Metric namespace | `BiomedicalPipeline` |
| CloudWatch | Publish frequency | 60 seconds |
| CloudWatch | Retention | 15 months |
| AWS Budgets | Alerts | 80% and 100% of $10/month |

All IAM roles follow the least-privilege principle: each component has a dedicated role with only the permissions strictly required for its function.
