# Distributed cloud architecture for near real-time biomedical data processing

Final Degree Project (TFG) — Data Engineering, Escola d'Enginyeria, Universitat Autònoma de Barcelona (UAB)  
**Author:** Lucía Revaliente Torres

---

## Overview

Distributed cloud architecture for the ingestion, processing, persistence and monitoring of biomedical data from wearable devices in near real-time. The use case is the continuous monitoring of hospital patients in non-critical areas.

The system receives continuous streams of physiological signals (ECG, EDA, BVP, TEMP, RESP, ACC), processes them with end-to-end latencies in the order of seconds, and stores both raw and processed records for traceability and analysis.

**Target SLO:** P95 pipeline latency < 10 seconds (measured from Kinesis arrival to end of Lambda processing).

---

## Architecture

Fully serverless, event-driven architecture on AWS, structured in five functional blocks:

```
Data Generator → Ingestion → Processing → Persistence → Monitoring
   (local)        Kinesis      Lambda      S3 + DynamoDB  CloudWatch
```

| Component | Service |
|---|---|
| Ingestion | Amazon Kinesis Data Streams |
| Processing | AWS Lambda + SQS DLQ |
| Raw persistence | Amazon S3 |
| Processed persistence | Amazon DynamoDB |
| Monitoring & audit | Amazon CloudWatch + AWS CloudTrail |
| Infrastructure | Terraform |

## Data model

Based on the [WESAD dataset](https://archive.ics.uci.edu/dataset/465/wesad+wearable+stress+and+affect+detection) (UCI ML Repository) as a structural reference for schema and sampling frequencies.

| Device | Sensor | Signal | Frequency |
|---|---|---|---|
| RespiBAN | ECG, EDA, EMG, TEMP, RESP, ACC | Various | 700 Hz |
| Empatica E4 | BVP | Pulse volume | 64 Hz |
| Empatica E4 | ACC | Acceleration | 32 Hz |
| Empatica E4 | EDA, TEMP | Electrodermal, temperature | 4 Hz |

Two record types flow through the pipeline:
- **Raw record** — event as generated by the simulator and ingested into Kinesis, written to S3 as-is for traceability.
- **Processed record** — enriched with three timestamps and three latency metrics, persisted to DynamoDB.

### Latency measurement model

All timestamps are generated within the AWS domain to avoid cross-machine clock skew. The data generator runs locally and its clock is intentionally excluded from measurements.

| Timestamp | Field | Source |
|---|---|---|
| T0 | `kinesis_arrival_ts` | `approximateArrivalTimestamp` assigned by Kinesis |
| T1 | `processing_start_ts` | `time.time()` at the start of per-event processing in Lambda |
| T2 | `processing_end_ts` | `time.time()` just before `put_item` to DynamoDB |

| Metric | Formula | Description |
|---|---|---|
| `kinesis_to_lambda_ms` | (T1 − T0) × 1000 | Time the event spent buffered in Kinesis before Lambda started processing it |
| `processing_latency_ms` | (T2 − T1) × 1000 | Internal Lambda processing time: decode, validate, enrich |
| `pipeline_latency_ms` | (T2 − T0) × 1000 | **End-to-end latency — primary SLO metric** |

> **Known limitation:** the DynamoDB `put_item` operation (~5–20 ms) is excluded from `pipeline_latency_ms`. This systematic underestimation represents at most 0.2% of the 10,000 ms SLO threshold and is considered negligible.

## Experimental scenarios

| Scenario | Subjects | Rate | Duration | Criterion |
|---|---|---|---|---|
| Base | 1 (Empatica E4, low freq.) | ~8 ev/s | 5 min × 3 | P95 < 10s |
| Sustained load | 2 (Empatica E4, all sensors) | ~208 ev/s | 5 min × 3 | P95 < 10s |
| Peak | 1 Empatica E4 + 1 RespiBAN | ~4,304 ev/s | 30s × 3 | No data loss, recovery < 60s |

Each scenario runs 3 consecutive repetitions with 10s between them. The `run_id` field (1, 2 or 3) identifies each repetition in DynamoDB.
